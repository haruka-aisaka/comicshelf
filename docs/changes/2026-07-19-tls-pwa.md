# TLS (HTTPS) 対応と PWA インストール要件の充足

## 背景

Android から HTTP でアクセスしているため、ブラウザが PWA と認識せず、
「ホーム画面に追加」の自動サジェスト (beforeinstallprompt) が発火しない。

Chrome の PWA installability 判定には以下が必要:

1. HTTPS (localhost 以外) でのアクセス
2. Service Worker が有効かつ fetch ハンドラを持つ (現状 OK)
3. `manifest.json` に **192x192** と **512x512** の PNG アイコン (any/maskable)
4. `name` / `short_name` / `start_url` / `display: standalone` (現状 OK)

現状 1 と 3 を満たしていないため、両方をこの変更で解消する。

## 現状の挙動

- comicshelf は `http://100.73.132.118:8080` (Tailscale IP) でプレーン HTTP 待ち受け。
- Tailscale Serve は既に稼働中だが、443 / 8443 / 8543 は他サービスが占有済み:
  - `443/` → `127.0.0.1:3457` (別サービス)
  - `8443/` → `127.0.0.1:8480`
  - `8543/` → `localhost:5984`
- Android Chrome でアクセスしても、
  - PWA インストールのサジェストが出ない
  - 「アプリをインストール」メニューがそもそも表示されない
- manifest のアイコンは `icon.svg` (1 個) のみ登録。

## 変更後の挙動

### アクセス URL

- Tailscale Serve に **空きポート (8444) の HTTPS エントリを追加** し、
  `https://raspberrypi.tailc8fdcd.ts.net:8444/` からアクセス可能にする。
  - 既存の 443 / 8443 / 8543 マウントには一切触れない。
- 従来の `http://100.73.132.118:8080` は残す (デスクトップ等の既存導線を維持)。
  PWA として使いたい場合のみ HTTPS 側にアクセスする。
- 証明書は Tailscale が Let's Encrypt から自動発行するため、
  Android で「有効な証明書」として扱われる。
- 公開範囲は Tailnet 内のみ (Funnel は使わない)。

### PWA インストール

- Android Chrome で HTTPS URL にアクセスすると、
  数秒〜数十秒の閲覧後に「アプリをインストール」バナー / メニューが
  自動で表示されるようになる (Chrome のヒューリスティックに依存)。
- インストール後はスタンドアロン (アドレスバーなし) で起動する。
- 非標準ポート (8444) でも Chrome の installability チェックは通る (HTTPS であれば可)。
- iOS Safari は従来通り「ホーム画面に追加」経由 (自動サジェストなし)。

### アイコン

- 192x192 / 512x512 の PNG を `web/public/icons/` に追加。
- `manifest.json` の `icons` 配列に SVG に加え PNG を登録:
  - `icon-192.png` (any)
  - `icon-512.png` (any)
  - `icon-512-maskable.png` (maskable)
- 既存 SVG も残し、対応環境では優先される。

### Service Worker

- Cache version を bump (`comicshelf-v21` → `v22`)。
  新しい manifest / アイコンを再取得させるため。
- SW の origin は URL に紐づくため、`https://*.ts.net:8444` と旧 `http://100.73.132.118:8080`
  は別 origin のキャッシュとして扱われる (仕様通り)。

## スコープ外

- Tailscale Funnel によるインターネット公開 (Tailnet 内のみ)。
- 独自ドメイン対応。
- iOS の PWA 挙動改善 (Safari の仕様に依存するため今回は触らない)。
- アプリ側のコード変更 (server.ts / routes / config)。
- 既存 443 / 8443 / 8543 マウントへの介入。
- 旧 HTTP エンドポイントの廃止。

## 実装メモ (参考、UX には影響しない)

- `tailscale serve --bg --https=8444 http://100.73.132.118:8080` をホスト側で実行。
  - `--bg` で永続化 (再起動後も維持)。
  - 既存の Serve エントリには影響しない (add-only)。
- アイコンは既存 `icon.svg` から `rsvg-convert` / ImageMagick で生成。
- compose.yaml は変更しない (Tailscale Serve は docker 外で動作)。

## E2E テストチェックリスト

### 正常系

- [ ] `https://raspberrypi.tailc8fdcd.ts.net:8444/` に Android Chrome でアクセスし、
      鍵マークが有効 (証明書エラーなし) であることを確認
- [ ] 同 URL を数十秒閲覧後、Chrome のメニューまたはバナーに
      「アプリをインストール」が現れる
- [ ] インストール後、ホーム画面のアイコンから起動するとスタンドアロン表示になる
- [ ] Chrome DevTools (PC で `chrome://inspect` 経由) の
      Application → Manifest で "Installability" にエラーがない
- [ ] 従来の `http://100.73.132.118:8080` でも従来通り閲覧できる
- [ ] 既存の Tailscale Serve マウント (443, 8443, 8543) が従来通り動く

### 異常系

- [ ] Tailscale が停止しているデバイスから `*.ts.net:8444` にアクセスすると
      到達不能になる (Tailnet 内限定である旨の確認)
- [ ] 証明書取得前 (Serve 追加直後) にアクセスした場合、
      Tailscale の自動リトライで数秒〜数十秒後に接続可能になる

### エッジケース

- [ ] SW キャッシュに古い manifest が残っていても、Cache version bump により
      次回ロード時に新 manifest / PNG アイコンが取得される
- [ ] インストール済み PWA を開いた状態で SW を更新した場合、
      リロードで新バージョンが反映される
- [ ] `https:8444` 側と `http:8080` 側は別 origin のため、
      それぞれ独立に SW が登録される (相互に干渉しない)
- [ ] iOS Safari で `https:8444` URL にアクセスした場合、
      「ホーム画面に追加」からインストールでき、
      アイコンが 512 PNG になる
