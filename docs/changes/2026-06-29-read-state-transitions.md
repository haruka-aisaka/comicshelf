# 既読ステータスの遷移ルールを明確化する

## 背景

既読の本を開き直して途中まで戻ると、 現在は意図せず「読書中」に降格して
しまう。 これは viewer の `saveProgress` が呼ばれるたびに
`finished = (currentPage >= totalPages - 1)` を再計算し、 末尾以外では
`finished: false` を送ってしまうためで、 既読の事実が再読操作で消える形に
なっている。

加えて、 既読本を開いた時の初期位置が「最終ページ」になっており、 再読を
始めるには毎回ユーザーが手動で先頭まで戻す必要がある。

「読み終わった」は事実として残るべき情報であり、 再読のたびに失われないよう
にしたい。 同時に、 ステータス遷移の起点をユーザーの明示操作に揃え、 不意な
ステータス変更を起こさないルールに整える。

## 現状の挙動

### 既読化のトリガ

- ビューワーで最終ページに到達すると、 `saveProgress` が
  `finished: true` を送信し自動的に既読化される
- ビューワーメニューの「既読にして閉じる」ボタンでも既読化される

### 既読本を再度開いた時

- `viewer.js` が `bookData.readState.lastPage` から再開するため、 既読本は
  **最終ページから** 表示される
- そこからページを戻すと、 ペアのページ送り (`scheduleSave`) で
  `finished: false` が送信され、 **既読が読書中に降格** する
- サーバ側 (`upsertReadState`) は patch の `finished` をそのまま採用する
  ため、 一度立った既読フラグも降格できてしまう

### 未読に戻す操作

- 提供されていない (UI 経路なし)。 DB を直接操作する以外に既読を取り消せない

## 変更後の挙動

### 既読化のルール

既読フラグは **ユーザーの明示操作でのみ on/off** する。 ページ位置の変化や
最終ページ到達では自動で書き換わらない。

具体的な遷移ルール:

- **読書中 → 既読**: 最終ページで「次へ」操作 (タップ / スワイプ / 矢印キー /
  `#next` ボタン) を行った時にモーダルを表示し、 「既読にして閉じる」を
  押した場合のみ既読化
  - メニューの「既読にして閉じる」ボタンからも従来通り既読化可能
- **既読 → 読書中**: 提供しない。 既読を解除したい場合は「未読に戻す」 (後述)
- **既読 → 未読**: ビューワーメニューの「未読に戻す」ボタンから明示的に。
  `lastPage = 0` / `finished = false` にリセット

### 進捗保存 (`saveProgress`) の挙動

- ページ移動時の `saveProgress` は **`lastPage` のみ送信**。 `finished` は
  送らない (= サーバ側で既存値を維持)
- ただし **既読本では `saveProgress` を呼ばない** (再読中の途中位置で
  `lastPage` を上書きしない。 再度開いた時にまた先頭から始められるように
  するため)

### 最終ページでのモーダル

未読 / 読書中の本で最終ページに到達した状態で「次へ」を試みた時:

- モーダル (中央に小さいダイアログ + 背景 backdrop) を表示
- メイン CTA: **「既読にして閉じる」** (緑系 / プライマリ) → POST
  `{ lastPage: totalPages-1, finished: true }` → トップへ遷移
- セカンダリ: **「キャンセル」** (グレー) → モーダルを閉じてビューワーに戻る
  (進捗は変えない)

発火条件の詳細:

- **明示的な「次へ」操作の時だけ** 発火する (タップ / スワイプ / 矢印キー /
  `#next` ボタン)
- シークバーや「ページ番号入力」で最終ページに飛んだだけでは発火しない
  (誤発火防止)
- **自動ページ送り (auto-advance)** で最終ページに到達 → さらに 1 拍進んだ
  タイミングで発火し、 同時に auto-advance を停止する
- **既読本でも同じモーダルを出す**。 「閉じる」動線として再利用 (`finished:
  true` の再送信は idempotent でステータス変化なし、 トップへ遷移する効果
  だけが意味を持つ)。 メニューを開いて閉じる手間を省く目的

### 既読本を開いた時の初期表示位置

- 既読本は **先頭ページ (page=0)** から開く
- `lastPage` の値は読了時のもの (= 最終ページ) のまま DB に保持する
  (履歴として残す)
- URL に明示的に `?page=N` がある場合は従来通りそれを優先

### ビューワーメニューの「未読に戻す」ボタン

メニューシート末尾 (現在「既読にして閉じる」がある位置) のボタンを、 読書
状態によって出し分ける:

| 状態     | ボタン文言              | 押下時の挙動                                                   |
| -------- | ----------------------- | -------------------------------------------------------------- |
| 未読     | (ボタン非表示)          | -                                                              |
| 読書中   | 「既読にして閉じる」    | `finished: true` で保存 → トップへ                             |
| 既読     | 「未読に戻す」          | 確認ダイアログ → `lastPage: 0, finished: false` で保存 → 留まる |

- 「未読に戻す」は破壊的操作 (lastPage を失う) なので確認ダイアログを挟む:
  「この本を未読に戻しますか? 読書位置の記録はリセットされます。」 →
  「未読に戻す」 / 「キャンセル」
- 「未読に戻す」確定後はトップに遷移せず、 ビューワーに留まる (ユーザーが
  再読のためにメニューを開いたケースを想定)

### 一覧側の表示

- ステータスバッジは既存の判定ロジック (`status` 派生) で十分。 今回のルール
  変更で「既読」「読書中」「未読」の定義自体は変わらない
- 「最近読んだ」 (`finished=1` の更新日時 DESC) は引き続き有効。 既読本を
  開いて先頭から見ても `finished=1` 維持なのでセクションに残る

### API への影響

`POST /api/books/:id/progress` のサーバ実装を以下のように変更:

- `finished` フィールドが **省略された場合は既存値を維持** する (現状もこの
  挙動だが、 クライアントが毎回送らなくなるため明示)
- `finished: true` への遷移は明示的にクライアントから送られた時のみ
- `finished: false` への遷移は「未読に戻す」操作からのみ (lastPage=0 と
  セットで送信)

API スキーマ自体は変更なし。 クライアントの送信パターンを変えるのが主。

## スコープ外

- 再読の進捗を別途記録する仕組み (「読了履歴」「読了回数」など) → 将来検討
- 既読本を開いた時に「先頭から / 続きから」を毎回問う UI → 今回は「先頭から
  固定」に倒す
- 自動既読化のオプション化 (設定で「最終ページ到達で自動既読」をオンに
  戻せるようにする等) → 提供しない
- 既読バッジを一覧から長押ししての「未読に戻す」 → 今回は導入しない
  (ビューワーメニュー一択)
- シリーズ / フォルダ単位の一括ステータス操作

## E2E テストチェックリスト

### 正常系

- [ ] 未読の本を開いて最終ページまで「次へ」操作で進むとモーダルが表示され、
      「既読にして閉じる」で既読化されてトップに戻る
- [ ] 同モーダルで「キャンセル」を押すとビューワーに戻り、 `finished` は
      変化しない (依然として読書中 or 未読)
- [ ] 既読の本を開くと先頭ページ (page=0) から表示される
- [ ] 既読本でページを進める / 戻す操作をしても DB 上の `lastPage` /
      `finished` は変わらない
- [ ] 既読本の最終ページで「次へ」操作するとモーダルが出て、 「既読にして
      閉じる」でそのままトップへ遷移する (閉じる動線として再利用)
- [ ] ビューワーメニューで既読本は「未読に戻す」ボタンが表示され、 確認
      ダイアログを経て押下すると `lastPage=0 / finished=false` になる
- [ ] 「未読に戻す」確定後はビューワーに留まる (トップ遷移しない)
- [ ] 「未読に戻す」確認ダイアログで「キャンセル」すると状態は変わらない
- [ ] 読書中の本のメニューには「既読にして閉じる」が表示され、 既読本では
      「未読に戻す」に切り替わる (未読本ではどちらも非表示)
- [ ] 既読化された本は一覧の「最近読んだ」セクションに表示される
- [ ] 既読本を開いて先頭から見て閉じても、 「最近読んだ」から消えない

### 異常系

- [ ] `POST /progress` で `finished` を送らない (`{lastPage: N}` のみ)
      リクエストは既存 `finished` を維持する
- [ ] 既読本に対して `{lastPage: K, finished: false}` を直接送れば未読化
      できる (API レベルでは引き続き許可。 UI からは「未読に戻す」のみ)
- [ ] ネットワーク失敗時、 「既読にして閉じる」「未読に戻す」は toast 表示
      して状態を戻す

### エッジケース

- [ ] 自動ページ送り中に最終ページに到達 → 1 拍後にモーダル表示、 同時に
      auto-advance が停止する
- [ ] シークバー / ページ番号入力で最終ページに飛んだだけではモーダルが出ない
- [ ] スプレッド (見開き) 表示時、 見開きの右ページが最終ページのときに
      「次へ」を試みるとモーダルが出る (`currentPage + 2 > totalPages - 1`
      でも発火)
- [ ] 既読本に `?page=N` で直接アクセスすると先頭でなくその位置から開く
- [ ] 既読本を開いて先頭から読み進めている途中で別画面に遷移して戻ると、
      また先頭から始まる (lastPage は更新されないため)
- [ ] BFCache で戻ったときも上記ルールが維持される (saveProgress の挙動が
      page 表示後に走らない)
- [ ] iOS Safari でモーダル表示中に backdrop タップ / Esc キーで「キャンセル」
      相当の挙動になる
- [ ] モーダル表示中はメインの seek / タップ操作が遮断される
- [ ] 「未読に戻す」直後にすぐ最終ページまで「次へ」操作すると、 通常通り
      モーダルが出て既読化できる (状態がきれいにリセットされている)

## Done 判定基準

- [ ] `saveProgress` が `finished` を送らなくなり、 サーバは送られなかった
      場合に既存値を維持する
- [ ] 既読本では `saveProgress` (ページ移動時の自動保存) が呼ばれない
- [ ] 最終ページでの「次へ」操作でモーダルが出て、 既読化はそこからのみ
- [ ] 既読本は先頭から開かれる
- [ ] ビューワーメニューが読書状態に応じて「既読にして閉じる」/「未読に戻す」
      を出し分ける
- [ ] 「未読に戻す」は確認ダイアログ経由で、 確定後はビューワーに留まる
- [ ] 上記 E2E チェックリストがすべて PASS
