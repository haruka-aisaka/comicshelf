# ビューワーの戻るボタンで検索状態を保持

## 背景

タグ/作者で絞り込んだ後に本を開き、 viewer の「←」 ボタンで一覧に戻ると
**検索バーが空になり、 絞り込みが解除される**。 ブラウザの戻る (システムボタンや
スワイプジェスチャー) では q が保持されるが、 UI 上の「←」 だけが捨てている。

## 現状の挙動

- viewer.html の左上に `<a href="/" class="back">←</a>` がある
- これは常に **ルート `/` への遷移** = query なし
- 結果: 検索/絞り込み中に viewer に入り、 「←」 で戻ると検索状態が失われる
- ブラウザの戻る (history.back) は同じ箇所に戻れる (q 保持) ので、 ユーザーは
  「UI ボタンの挙動が一貫していない」 と感じる

## 変更後の挙動

viewer の「←」 ボタンタップで:

1. **直前のページが同じサイト内なら history.back() を呼ぶ**
   - 検索や絞り込み・スクロール位置・展開済みのディレクトリツリーなど、
     一覧画面の状態が完全に保たれる
2. **直前のページがない場合 (URL 直開きやリロード後) は `/index.html` にフォールバック**
   - 履歴が空でも、 ボタンが反応しないことはない

判定方法:
- `document.referrer` が同一オリジンか、 `history.length > 1` を見る
- 両方 false なら fallback

## スコープ外

- 「N1: スクロール位置復帰」 は既に sessionStorage で実装済み。 history.back で
  自然に復元されるので変更不要
- 設定画面の「←」 ボタン (同じく `href="/"`)。 設定画面は通常 viewer から
  経由しないので影響なし (今回は触らない)
- ブラウザの戻るボタンの挙動 (既に正しい)

## E2E テストチェックリスト

### 正常系

- [ ] /?q=shayo で一覧開く → カード click で viewer → 「←」 で戻ると
      URL に `?q=shayo` が残り、 検索 input にも値が入っている
- [ ] グリッドが絞り込まれた状態 (件数が同じ) で表示される
- [ ] ディレクトリで絞り込んだ状態 + viewer → 「←」 で、 ディレクトリ絞り込みも保持
- [ ] /viewer.html?book=ID を **直接** 開いて 「←」 を押すと `/index.html`
      (= 通常の一覧) に遷移する

- [ ] 履歴がない viewer 直開きで 「←」 を連打しても安定動作

- [ ] 検索 + ディレクトリ + ステータスフィルタ複合状態でも全て保持
