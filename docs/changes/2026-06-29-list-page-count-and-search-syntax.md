# 一覧画面のページ数表示 + 検索クエリ構文の拡張

## 背景

- **ページ数表示**: 書籍を選ぶ前にページ数を知りたいケースが多い (短編か
  長編か、 続きを読むのに時間が取れるか等)。 現状は viewer に入ってから
  しか分からない。
- **検索の field 指定**: 現状は `?q=岸本` がタイトル / ディレクトリ /
  ComicInfo の主要 9 フィールドを横断 LIKE。 「writer が岸本のもの」を
  正確に絞りたい時に、 たとえば「岸本」というタイトルや「岸本」というタグが
  あるとノイズになる。 chip タップで作者で絞り込むときも、 同じ名前を
  含む別フィールドが混ざるので使いにくい。

## 現状の挙動

### ページ数表示
- ビューワー内のシークインジケータ (`current / total`) でのみ確認可能
- 一覧画面のカードにはページ数を出していない

### 検索
- 検索バーは全文字列を 1 つの LIKE パターンに変換し、 タイトル / ディレクトリ /
  ComicInfo の `title` / `series` / `writer` / `penciller` / `imprint` /
  `publisher` / `characters` / `tags` / `genre` を OR 結合で検索
- カードの作者 chip / ビューワーのタグ chip タップは `?q=<値>` で
  この横断 LIKE と同じ挙動
- 全マッチは LIKE `%値%` の部分一致 (大文字小文字無視)

## 変更後の挙動

### 一覧カードのページ数表示

カード下部の「作者」 メタ行を拡張し、 ページ数を併記する。

```
[ サムネ ]
タイトル
作者名 ・ 123p          ← 作者がある場合 (・ で区切る)
                         作者がない場合: 123p のみ
                         pageCount=null のとき: 何も出さない
```

- セクション (続きから / 最近追加した / 最近読んだ) のカードでも同じ
- カード幅が狭い時はメタ行のテキストを ellipsis で省略 (既存スタイル踏襲)
- ページ数表示は ComicInfo の `pageCount` ではなく **`books.page_count`**
  (実ファイルから抽出した数) を使う。 ComicInfo の値とずれていても、
  実際に開ける枚数を信用する

### 検索クエリ構文 (field prefix)

検索バー / chip タップ / URL `?q=` で、 以下の prefix を解釈する:

| prefix | 対象フィールド |
| --- | --- |
| `tag:<値>` | ComicInfo の `tags` (JSON 配列) に `<値>` が要素として含まれる |
| `genre:<値>` | ComicInfo の `genre` (JSON 配列) に `<値>` が要素として含まれる |
| `writer:<値>` | ComicInfo の `writer` が `<値>` に完全一致 |
| `penciller:<値>` | ComicInfo の `penciller` が `<値>` に完全一致 |
| `series:<値>` | ComicInfo の `series` が `<値>` に完全一致 |
| `publisher:<値>` | ComicInfo の `publisher` が `<値>` に完全一致 |
| `imprint:<値>` | ComicInfo の `imprint` が `<値>` に完全一致 |
| `character:<値>` | ComicInfo の `characters` (カンマ区切り) に `<値>` が含まれる |
| (prefix なし) | 従来通りタイトル / ディレクトリ / ComicInfo 主要フィールドを
                  横断 LIKE 部分一致 |

#### マッチング規則

- すべて大文字小文字無視 (`COLLATE NOCASE`)
- 配列フィールド (tags / genre / characters): 要素単位で完全一致
  (例: `tag:アクション` は要素 "アクション" にマッチ。 "アクションヒーロー"
  にはマッチしない)
- 単一フィールド (writer / series 等): 値全体に完全一致

#### 複数指定の AND 結合

クエリ内に複数の prefix トークンを並べると、 それぞれが AND で結合される:

```
writer:岸本斉史 tag:忍者         → writer が岸本斉史 AND tags に "忍者"
series:NARUTO writer:岸本斉史   → series が NARUTO AND writer が岸本斉史
```

prefix なしのトークンが混ざる場合は、 prefix なし部分は従来の横断 LIKE
として AND される:

```
writer:岸本 ナルト
  → writer が "岸本" に完全一致 AND "ナルト" が
    タイトル / ディレクトリ / ComicInfo 主要フィールドのいずれかに部分一致
```

#### トークン化規則

- 半角スペースでトークンを区切る
- 値にスペースを含めたい場合は `"..."` で引用 (例: `writer:"Akira Toriyama"`)
- `prefix:` の直後が引用符でなければ、 次のスペースまでが値
- 不明な prefix (例: `foo:bar`) は **prefix なしのトークン扱い** で
  従来通り横断検索にフォールバック (エラーにしない)
- 値内のコロンはエスケープ不要 (`writer:岸本:斉史` のように指定したい
  人はいない想定。 ある場合は引用符で包む `writer:"a:b"`)

#### URL への反映

- 検索バーの入力はそのまま `?q=` に乗る
- chip タップ (カードの作者 / ビューワーのタグ・シリーズ・作者) は、
  対応する prefix 付きの値に書き換えて `?q=writer:岸本斉史` のような
  URL を生成する
- 既存の chip タップで `?q=岸本斉史` を直接生成していた挙動は廃止
  (= prefix 付きが新しい標準)

#### 検索バーの placeholder ヒント

```
タイトル / writer:作者 / tag:タグ ...
```

を入れて、 構文があることに気付けるようにする。 文字列定数のみで
docs リンクは出さない (シンプルさ優先)。

### サーバー側 API

- `GET /api/books?q=...` のクエリ文字列を、 サーバ側で上記構文に従って
  パース
- パース結果は内部の WHERE 句に複数 AND として乗せる
- 旧形式 (prefix なし全文) は引き続き動作

### ビューワー内 chip タップ

ビューワーのメタパネルの chip (シリーズ / 作者 / タグ) タップは、
それぞれ `series:` / `writer:` / `tag:` 付きの URL に変える。

## スコープ外

- 全文検索のスコアリング / ハイライト
- `OR` 結合構文 (今回は AND のみ)
- `NOT` 否定構文 (`-writer:...` 等)
- 数値範囲 (`pages:>100` 等)
- ページ数の取得 (バックエンドは既に書籍の page_count を持っているので
  追加スキャンは不要)
- ページ数表示のフォーマット切り替え (常に "123p")
- writer の autocomplete / 候補表示
- 引用符の中の `\"` エスケープ (まずは単純 split)
- chip タップ以外の URL 生成箇所への影響 (ディレクトリツリーは無関係)

## E2E テストチェックリスト

### 正常系

- [ ] ComicInfo の pageCount が分かっている書籍のカード下部に
      「作者 · 123p」 が表示される
- [ ] 作者が無い書籍では「123p」 のみが表示される
- [ ] `?q=writer:岸本斉史` で writer が完全一致する書籍だけが出る
- [ ] `?q=tag:忍者` で tags 要素に「忍者」 を含む書籍だけが出る
- [ ] `?q=writer:岸本斉史 tag:忍者` で両方を満たす書籍だけが出る
- [ ] `?q=writer:岸本 ナルト` で writer 完全一致 + ナルト部分一致の
      AND 結合になる
- [ ] カードの作者 chip タップで URL が `?q=writer:<author>` に変わり、
      一覧がその作者のものだけに絞られる
- [ ] ビューワーのタグ chip タップで `/?q=tag:<tag>` に遷移する
- [ ] ビューワーのシリーズ chip タップで `/?q=series:<series>` に
      遷移する
- [ ] 検索バーの placeholder に構文ヒントが表示される

### 異常系

- [ ] 未対応 prefix (`foo:bar`) は無視され、 トークン全体が prefix なし
      として横断検索にフォールバックする
- [ ] `tag:` のみ (値なし) のトークンは無視される
- [ ] 引用符が閉じていない場合 (`writer:"未閉じ`) はエラーにせず、
      引用符以降を値の続きとして処理する
- [ ] 値に LIKE メタ文字 (`%` / `_`) が含まれていても、 完全一致は
      影響を受けない

### エッジケース

- [ ] pageCount が 0 の書籍 (空 ZIP 等) は `0p` ではなく表示しない
      (= null 扱いと同じ)
- [ ] writer が空文字 ("") の書籍に対して `writer:""` で完全一致できる
      (使うかは別として、 仕様として一貫している)
- [ ] 配列フィールド (tags) の値が NULL の書籍は `tag:...` ではヒット
      しない
- [ ] 検索バーに `writer:` のみ入力した状態 (= 値なし) は、 全件表示に
      フォールバック (= 検索条件が無いとみなす)
- [ ] 値に space を含む引用形式 (`writer:"Akira Toriyama"`) が正しく
      認識される
- [ ] 大量に prefix を並べた場合 (5 つ以上) もパフォーマンス劣化が
      ないこと (SQL は AND 条件追加のみで indexed columns に効く)
- [ ] BFCache 復元時に検索バーの値が URL から再同期され、 prefix 付き
      でも維持される

## Done 判定基準
